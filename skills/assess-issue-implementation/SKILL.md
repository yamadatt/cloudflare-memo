# ISSUE実装検討

この手順で、Issueに対して実装すべきかを判断し、実装する場合は3パターンの案を提示する。
操作は GitHub CLI（`gh`）を使い、MCPは使わない。

## 手順

1. 対象Issueを確定する。
- Issue番号またはIssue URLを確定する。
- スコープ外の課題と混同しないように、タイトル・ラベル・担当情報を確認する。

2. Issueコンテキストを収集する。
- `scripts/collect_issue_context.sh <issue-number>` を実行する。
- 本文、コメント、ラベル、関連PR、更新履歴を確認する。

3. 実装要否を判断する。
- 目的が明確か、既存機能で代替可能か、運用回避が可能かを確認する。
- 結論を `実装する / 実装しない / 追加調査` の3択で出す。
- 「実装しない」の場合でも、根拠と代替策を明記する。

4. フィージビリティを評価する。
- 技術面: 既存アーキテクチャ適合性、依存関係、データ影響。
- 工数面: 実装、テスト、移行、運用負荷。
- リスク面: 回帰、セキュリティ、性能、障害時の影響。
- 各観点を `高 / 中 / 低` で評価し、理由を1行で付ける。

5. 実装案を3パターン作成する。
- A案（最小）: 変更量最小・短納期重視。
- B案（標準）: バランス重視（基本推奨）。
- C案（拡張）: 将来拡張性重視。
- 各案に「概要」「変更箇所」「メリット」「デメリット」「概算工数」「主なリスク」「ロールバック」を記載する。

6. 推奨案と次アクションを提示する。
- 3案比較のうえ推奨案を1つ選び、採用理由を明記する。
- 次アクションとして、実装タスク分解（例: 設計/実装/テスト/デプロイ）を短く提示する。

7. Issueへ必ず投稿する（省略不可）。
- 調査結果（Issue本文/コメント/関連PRの要点）と検討結果（実装要否、3案、推奨案、次アクション）をMarkdownにまとめる。
- `gh issue comment <issue-number> --body-file <file>` を必ず実行し、Issueコメントへ追記する。
- 投稿後はコメントURLを取得し、ユーザーへの返答に含める。

## 出力形式

以下の順で返す:
1. 実装要否の結論（3択）
2. フィージビリティ評価（技術/工数/リスク）
3. 実装案3パターン（A/B/C）
4. 推奨案と採用理由
5. 次アクション
